# Wheeler: Architecture

## Four Layers

Wheeler is four things stacked:

```
+---------------------------------------------+
|  ACTS          /wh:* slash commands         |  What you DO
|                bin/wh headless runner        |
+---------------------------------------------+
|  FILE SYSTEM   .notes/*.md (prose)          |  What you KNOW
|                .plans/*.md (state)           |
|                docs/, scripts/ (artifacts)   |
+---------------------------------------------+
|  SYNTHESIS     synthesis/*.md (Obsidian)    |  How you BROWSE
|                YAML frontmatter + backlinks  |
+---------------------------------------------+
|  GRAPH         knowledge/*.json (index)     |  How things CONNECT
|                metadata + relationships      |
|                embeddings + file pointers     |
+---------------------------------------------+
```

**Acts** are slash commands, the verbs. `/wh:discuss`, `/wh:plan`, `/wh:execute`, `/wh:write`, `/wh:add`, `/wh:compile`, `/wh:dream`, and more. Each one gives Claude the right tools and constraints for that stage of work. YAML frontmatter restricts which tools are available per mode. The markdown body is the system prompt. No custom orchestration code: Claude Code is the orchestrator.

**File system** is where real artifacts live. Research notes are markdown in `.notes/`. Drafts and docs live wherever the scientist puts them. Scripts live in the project. Investigation state lives in `.plans/`. These are the actual work products: natural files you can browse, grep, git-diff, and read directly.

**Synthesis** is the human-browsable layer. `synthesis/*.md` files are Obsidian-compatible markdown with YAML frontmatter and `[[backlinks]]`. Auto-generated by the triple-write system alongside JSON metadata. Per-node files are created on every mutation; index files (`INDEX.md`, `OPEN_QUESTIONS.md`, `EVIDENCE_MAP.md`) are rebuilt by `/wh:dream`.

**Graph** is the index layer. `knowledge/*.json` files store node metadata (id, type, tier, timestamps). The graph database stores relationships (which finding came from which analysis, which paper informed which method), embeddings (for semantic search), and file pointers (path to the actual artifact). Graph nodes point to files. They don't contain prose.

The graph is a library catalog. You don't read the catalog: you read the books. The catalog tells you which books are related.

### Why Slash Commands, Not Agent SDK

Claude Code's native slash command system does everything:
- Mode enforcement via YAML `allowed-tools`
- System prompts via markdown body
- Context injection via `graph_context` MCP tool

This replaced ~1500 lines of planned Python orchestration.

### Headless Work

For background tasks: `wh queue "task"` and `wh quick "task"` run `claude -p` (headless) with structured JSON output. No API keys: runs on Max subscription.

## File System

The file system has two halves: the **scientist's workspace** (their data, code, writing) and **Wheeler-managed directories** (graph index, investigation state, notes).

### Scientist's Workspace

Wheeler doesn't own or copy files. It tracks where they are. Paths can point anywhere: local directories, external drives, network mounts, shared NAS:

```yaml
# wheeler.yaml
paths:
  code: ["scripts/", "/shared/lab/analysis-tools/"]
  data: ["data/", "/Volumes/LabNAS/recordings/"]
  results: ["results/"]
  figures: ["figures/"]
  docs: ["docs/"]
```

`/wh:ingest` scans these paths, creates Dataset and Script nodes that point to the actual files. `/wh:execute` knows where to find scripts and data without the scientist spelling it out. The graph stores paths as-is and validates with hashes.

### Wheeler-Managed

```
knowledge/      # Graph index: JSON metadata, one per node
synthesis/      # Human-readable: Obsidian-compatible markdown, one per node + indexes
.notes/         # Research notes: markdown from /wh:note and /wh:compile
.plans/         # Investigation state, plans, summaries
.logs/          # Headless task output
.wheeler/       # Internal (embeddings, request logs, etc.)
```

### Graph Metadata (`knowledge/`)

JSON files, one per knowledge node. Structured data the graph indexes:

```json
{
  "id": "F-3a2b1c4d",
  "type": "Finding",
  "tier": "generated",
  "description": "Calcium oscillation frequency scales with cell density...",
  "confidence": 0.85,
  "path": "figures/fig2a.png",
  "artifact_type": "figure",
  "source": "",
  "stability": 0.3,
  "stale": false,
  "created": "2026-03-26T14:30:00+00:00",
  "updated": "2026-03-26T14:30:00+00:00",
  "tags": []
}
```

Pydantic v2 models (`wheeler/models.py`) define the schema for all node types (including Script, Execution, and Ledger). `wh show F-3a2b` renders any node as readable markdown.

### Synthesis Files (`synthesis/`)

Obsidian-compatible markdown with YAML frontmatter and `[[backlinks]]`, auto-generated by the triple-write system. Each node gets a corresponding `.md` file. Relationships are rendered as backlinks.

```markdown
---
id: F-3a2b1c4d
type: Finding
tier: generated
created: 2026-03-26
confidence: 0.85
artifact_type: figure
---

# Finding [F-3a2b1c4d]

**Confidence**: 0.85 | **Tier**: generated | **Created**: 2026-03-26

Calcium oscillation frequency scales with cell density...

![figure](figures/fig2a.png)

## Relationships

- **WAS_GENERATED_BY** [[X-1234abcd]] (Script run: analyze_oscillations.py)
- [[D-5678efab]] **USED** this (Calcium recording dataset)
```

Index files (`INDEX.md`, `OPEN_QUESTIONS.md`, `EVIDENCE_MAP.md`) are rebuilt from scratch by `/wh:dream` Phase 3.5.

### Research Notes (`.notes/`)

Markdown files created by `/wh:note` and `/wh:compile`. The graph node in `knowledge/` points here via the node's content fields.

```markdown
---
id: N-4e5f6a7b
title: "Temperature dependence of calcium oscillations"
created: 2026-03-26
context: "Reviewing cell_042 recordings"
---

The oscillation frequency seems to drop when we cool the bath below 30C.
Could this be a channel gating effect?
```

### Investigation Files (`.plans/`)

YAML frontmatter + markdown body. Machine-readable metadata enables fast scanning.

| File | Created by | Purpose |
|------|-----------|---------|
| `STATE.md` | `/wh:init` | Current investigation state (<100 lines) |
| `{name}-CONTEXT.md` | `/wh:discuss` | Locked decisions, scope, success criteria |
| `{name}.md` | `/wh:plan` | Task list with waves, assignees, checkpoints |
| `{name}-SUMMARY.md` | `/wh:execute` | What happened, graph nodes created |
| `{name}-VERIFICATION.md` | completion | Citation audit, criteria met/unmet |
| `DREAM-REPORT.md` | `/wh:dream` | Consolidation results, promotions, flags |

### Logs (`.logs/`)

JSON output from headless `wh queue` / `wh quick` runs. Read by `/wh:reconvene`.

## Knowledge Graph

The graph stores **metadata and relationships**, not content. It is an index over the file system.

### What the Graph Stores

For each node:
- `id`, `type`, `tier`, `title` (~100 chars), `file_path`, `created`, `updated`
- `stability` (0.0-1.0, epistemic trust score), `stale` (boolean), `stale_since`
- `session_id`, `tags`
- Filterable metadata: `confidence` (Finding), `priority` (OpenQuestion), `status` (Hypothesis), `doi` (Paper)
- Type-specific fields: `path`, `artifact_type`, `source` (Finding), `hash` (Script), `kind` (Execution)
- Relationships to other nodes

For relationships (14 types):
```
PROV (W3C standard):
  USED, WAS_GENERATED_BY, WAS_DERIVED_FROM, WAS_INFORMED_BY,
  WAS_ATTRIBUTED_TO, WAS_ASSOCIATED_WITH

Semantic (Wheeler-specific):
  SUPPORTS, CONTRADICTS, CITES, APPEARS_IN, RELEVANT_TO,
  AROSE_FROM, DEPENDS_ON, CONTAINS
```

### What the Graph Does NOT Store

Full descriptions, statements, questions, or other prose content. That lives in the JSON files. The graph has a short `title` for context injection and display, but the file is the source of truth.

### Node Types

| Prefix | Type | Key Fields |
|--------|------|-----------|
| F | Finding | description, confidence, path, artifact_type, source |
| H | Hypothesis | statement, status (open/supported/rejected) |
| Q | OpenQuestion | question, priority (1-10) |
| D | Dataset | path, data_type, description |
| P | Paper | title, authors, doi, year (always tier=reference) |
| W | Document | title, path, section, status (draft/revision/final) |
| S | Script | path, hash, language, version |
| X | Execution | kind, agent_id, status, started_at, ended_at, description |
| PL | Plan | status |
| N | ResearchNote | title, content, context |
| L | Ledger | mode, prompt_summary, pass_rate, ungrounded, citations_* |

All node types inherit from `NodeBase`, which provides: `id`, `type`, `tier`, `created`, `updated`, `tags`, `stability`, `stale`, `stale_since`, `session_id`.

### Provenance Chain

```
Execution (kind: "script")
  -USED-> Paper (reference)
  -USED-> Script (path, hash)
  -USED-> Dataset (path, hash)
Finding -WAS_GENERATED_BY-> Execution
  -WAS_DERIVED_FROM-> Paper
  -APPEARS_IN-> Document
```

Script nodes store `hash` (SHA-256 at execution time), a cryptographic receipt of exactly what ran. Execution nodes record the activity (what ran, when, status). `detect_stale` re-hashes scripts and flags mismatches.

### Stability and Invalidation

Every node carries a `stability` score (0.0-1.0) encoding epistemic trust. Default values are assigned by type and tier:

| Node Type | Reference Tier | Generated Tier |
|-----------|---------------|----------------|
| Dataset | 1.0 | 0.7 |
| Paper | 0.9 | - |
| Finding | 0.8 | 0.3 |
| Script | 0.7 | 0.5 |
| Document | 0.7 | 0.3 |
| Hypothesis | 0.7 | 0.3 |

When a node's content changes (e.g., a script is edited), `propagate_invalidation()` in `wheeler/provenance.py` traverses downstream through PROV relationships using a transitive Cypher query:

```
(downstream)-[:WAS_GENERATED_BY|USED*2..N]->(changed)
```

Each entity-to-entity hop traverses 2 edges (WAS_GENERATED_BY + USED), so max_edges = max_hops * 2. Additionally, `WAS_DERIVED_FROM` chains are traversed separately. Stability decays exponentially per hop (default decay factor 0.8 = 20% reduction per hop). Affected nodes are marked `stale = true`.

`detect_and_propagate_stale()` combines hash-based staleness detection with downstream propagation: finds Script nodes whose file hash no longer matches disk, then propagates reduced stability through all transitive dependents.

### Context Tiers

Every node has `tier`: `reference` (established) or `generated` (new work). Papers are always reference. The `graph_context` tool separates findings by tier so Claude distinguishes what's verified from what's fresh.

### Backend

Neo4j is the only active backend, accessed via the `GraphBackend` ABC in `wheeler/graph/backend.py`.

- **Neo4j**: Docker or Neo4j Desktop, browser UI at `:7474`, raw Cypher via MCP `run_cypher` tool
- Connection: `bolt://localhost:7687`
- Selected by `config.graph.backend` (currently only `"neo4j"`)
- Factory: `get_backend(config)` returns `Neo4jBackend`

Community Edition supports per-project isolation via a `_wheeler_project` property on every node. When `config.neo4j.project_tag` is set, all MATCH/CREATE queries are scoped to that tag. Enterprise/Aura users get real database isolation.

### Semantic Search

Included by default (`fastembed` and `numpy` are core dependencies). Uses fastembed (BAAI/bge-small-en-v1.5) + numpy, stored in `.wheeler/embeddings/`. The `search_findings` tool uses multi-channel retrieval with Reciprocal Rank Fusion (RRF), combining semantic (embedding similarity), keyword (graph queries), and temporal (recency) channels for better recall than any single channel alone.

## How the Layers Connect

### Triple-Write: Act creates knowledge

Every `add_*` mutation writes three things:

```
/wh:execute
  -> runs analysis script
  -> calls add_finding (MCP tool)
  -> 1. Creates graph node in Neo4j
  -> 2. Writes knowledge/F-3a2b.json (atomic: tmp + rename)
  -> 3. Writes synthesis/F-3a2b.md (Obsidian: YAML frontmatter + backlinks)
  -> indexes embedding for semantic search
```

For `link_nodes`, `_update_synthesis_for_link()` re-renders synthesis files for both endpoints with updated Relationships sections. For `set_tier`, both JSON and synthesis files are updated. All synthesis writes are best-effort: if synthesis fails, graph and JSON are preserved.

### Act reads knowledge:
```
/wh:discuss
  -> calls graph_context (graph returns titles + tiers)
  -> Claude reads knowledge/F-3a2b.json for full content
  -> OR uses show_node MCP tool
```

### Act queries connections:
```
/wh:plan
  -> calls graph_gaps (graph finds unlinked questions, unsupported hypotheses)
  -> proposes investigations based on what's MISSING
```

### Act compiles synthesis:
```
/wh:compile "spike timing"
  -> searches graph for related nodes (multi-channel retrieval)
  -> traces provenance and relationships via Cypher
  -> writes .notes/compile-{topic}-{date}.md with citations
  -> registers Document node, links to all source nodes
```

## Layering Violations and Concerns

### 1. knowledge/migrate.py imports from graph layer (CROSS-LAYER)

`knowledge/migrate.py` has a top-level import of `wheeler.graph.backend.GraphBackend`. The documented architecture says knowledge/ depends only on models. However, this is architecturally justified: migration is inherently a bridge between the graph and filesystem layers. It reads from the graph backend and writes to knowledge files. This module is not part of the core knowledge store API; it is a one-time migration utility.

**Severity**: Low. The import is appropriate for the module's purpose.

### 2. validation/ledger.py imports from tools layer (UPWARD DEPENDENCY)

`validation/ledger.py` has a lazy import of `wheeler.tools.graph_tools.execute_tool` inside its `store_entry()` function. This is an upward dependency: validation (layer 3) reaches into tools (layer 3/4) to persist ledger entries.

**Severity**: Medium. The lazy import avoids circular dependency at load time. The coupling exists because ledger entries are stored as graph nodes via the same triple-write path as all other node types. This is intentional: it reuses the existing add_ledger tool rather than duplicating the triple-write logic. But it creates a hidden circular dependency chain: `tools.graph_tools.__init__` imports nothing from validation, but `validation.ledger` imports from `tools.graph_tools`.

**Potential fix**: Move `store_entry` to `tools/graph_tools/` or have ledger.py write directly via the backend + knowledge store, bypassing the tool dispatch.

### 3. validation/citations.py imports graph.schema at top level

`validation/citations.py` imports `PREFIX_TO_LABEL` from `graph.schema` at the top level. This is technically correct since both are in layer 2-3, but it means loading the validation module also loads the graph schema module. The graph driver and provenance imports are properly lazy (inside `validate_citations()`).

**Severity**: Low. `graph.schema` is lightweight (no I/O at import time).

### 4. tools/cli.py imports graph.driver at top level

`tools/cli.py` does `from wheeler.graph.driver import get_sync_driver` at module load time. This triggers `import neo4j`, which means the CLI fails to load if neo4j is not installed, even for commands that don't need it (like `install`, `version`, `show`).

**Severity**: Medium. The neo4j package is a core dependency in pyproject.toml, so this works in practice, but it's tighter coupling than necessary.

### 5. graph/context.py and graph/provenance.py use Neo4j driver directly

Both modules do top-level `from wheeler.graph.driver import get_async_driver`, bypassing the backend ABC. This is architecturally intentional: they use Neo4j-specific Cypher features (variable-length paths, complex aggregations) that the backend ABC does not abstract.

**Severity**: Low. Neo4j is the only active backend, so the bypass has no practical impact.

### 6. scaffold.py is dead code

`scaffold.py` has zero callers anywhere in the codebase. It was originally intended for project directory detection and creation but is unused. It still imports `config.py` and `yaml` at the top level.

**Severity**: Low. No functional impact, but it adds unnecessary weight to the module listing.

### 7. No circular import chains at load time

Despite the lazy upward dependency in `validation/ledger.py`, there are no circular imports at module load time. All potential cycles are broken by lazy imports inside functions. The import graph is a DAG at load time.

## Lazy Import Strategy

Wheeler uses lazy imports (inside functions) in four situations:

1. **Breaking potential cycles**: `validation/ledger.py` imports `tools.graph_tools.execute_tool` lazily to avoid a cycle between validation and tools layers.

2. **Optional dependencies**: `graph/backend.py` lazily imports `neo4j_backend` so only the selected backend's dependencies are loaded. `search/embeddings.py` lazily imports `fastembed`. `installer.py` lazily imports `packaging.version`.

3. **Deferring heavy imports**: `tools/cli.py` lazily imports `graph.trace`, `graph.provenance`, `installer`, `graph.backend`, and `knowledge` modules inside specific CLI commands so the CLI starts fast.

4. **Query enrichment**: `tools/graph_tools/queries.py` lazily imports `knowledge.store.read_node` inside each query function to enrich results with file data. `tools/graph_tools/__init__.py` lazily imports `knowledge.render.render_synthesis` and `provenance.default_stability` for triple-write.

## Design Principles

1. **Four layers, clear boundaries**: acts operate, files store, synthesis browses, graph connects
2. **Files are source of truth**: graph is an index, not a database
3. **Triple-write keeps layers in sync**: every mutation writes graph + JSON + synthesis
4. **Everything is a reference**: claims cite graph nodes `[F-3a2b]`
5. **Deterministic validation**: regex + Cypher, not LLM self-judgment
6. **Structure scales with presence**: loose when interactive, strict when independent
7. **Task routing**: scientist thinks, Wheeler grinds, never the reverse
8. **Provenance is cryptographic**: script hashes, not self-reports
9. **Stability tracks epistemic trust**: scores propagate through PROV chains
10. **No orchestration code**: Claude Code is the orchestrator
11. **No API calls**: runs on Max subscription, `claude -p` for headless

## Key Design Decisions

| Decision | Choice | Why |
|----------|--------|-----|
| Content storage | JSON files in `knowledge/` | Browsable, git-trackable, AI-readable, no query language needed |
| Browsable layer | Obsidian markdown in `synthesis/` | YAML frontmatter + backlinks, works in any editor or Obsidian vault |
| Graph role | Metadata + relationships only | Graphs are good at connections, bad as document stores |
| Orchestration | Claude Code slash commands | Zero Python orchestration code, YAML tool restrictions |
| Graph backend | Neo4j only | Browser UI at :7474, raw Cypher, project namespace isolation for Community Edition |
| Headless work | `claude -p` subprocess | No API key, Max subscription billing |
| Search | Multi-channel RRF (semantic + keyword + temporal) | Better recall than any single channel, graceful degradation |
| Embedding model | fastembed (BAAI/bge-small-en-v1.5) | 33MB model, no PyTorch, file-based persistence |
| Models | Pydantic v2 | Schema validation, JSON serialization, discriminated unions |
| Config | YAML (`wheeler.yaml`) | Human-readable, per-project |
| Stability scoring | Default by (type, tier) + propagation | Epistemic trust encoded as numbers, automatic staleness cascade |
| Request logging | Append-only JSONL (`.wheeler/request_log.jsonl`) | Latency tracking, error rates, session auditing |

## Architectural Recommendations

### Currently Sound

- **models.py and config.py are true leaf nodes** with zero internal dependencies. This is the right foundation.
- **knowledge/store.py and knowledge/render.py** depend only on models, as documented. Clean.
- **Lazy import strategy** is consistent and well-applied. No circular imports at load time.
- **Triple-write pattern** in tools/graph_tools/__init__.py ensures graph, filesystem, and synthesis stay in sync.
- **Multi-channel retrieval** in search/retrieval.py gracefully degrades when individual channels are unavailable.
- **Stability scoring** in provenance.py uses sensible defaults and exponential decay for propagation.

### Worth Watching

- **validation/ledger.py's upward dependency on tools** is the one true layering violation. It works via lazy import but makes the dependency graph harder to reason about.
- **fastembed/numpy as core deps** means every `pip install wheeler` pulls in the embedding model. This is intentional but worth documenting prominently for users who don't want search.
- **scaffold.py is dead code** with zero callers. It could be removed to reduce module count.
